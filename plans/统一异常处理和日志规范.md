# FastAPI 统一响应、异常处理、日志方案

## Summary

为当前 `movieclaw` 服务补一套轻量但可长期演进的基础设施，原则如下：

- 仅对“业务接口成功响应”使用统一包装结构
- 基础接口如 `/health` 保持轻量原生返回，不强制套壳
- 所有失败响应统一成同一错误结构
- 业务错误码使用字符串语义码
- 错误返回对外保持用户友好，校验类错误单独提供 `details`
- 日志先采用本地友好的文本格式
- 当前版本不引入 `request_id`

## Key Changes

### 1. 统一成功响应模型

新增通用响应模型，作为业务接口默认返回壳，结构固定为：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

实现约束：

- 新增一个通用 `ApiResponse[T]` 泛型模型，放在公共 schema 模块
- 提供一个响应构造辅助函数，例如 `ok(data, message="success", code="OK")`
- 后续所有业务路由优先返回该结构
- `/api/v1/health` 保持现有原生返回，不强制改成统一包装
- OpenAPI 中业务接口应声明统一响应模型，避免文档和真实返回不一致

### 2. 统一错误响应与异常层次

新增统一错误响应结构：

```json
{
  "success": false,
  "code": "VALIDATION_ERROR",
  "message": "request validation failed",
  "details": [...]
}
```

错误模型字段约定：

- `success`: 固定 `false`
- `code`: 字符串语义码，例如 `BAD_REQUEST`、`VALIDATION_ERROR`、`RESOURCE_NOT_FOUND`、`INTERNAL_SERVER_ERROR`
- `message`: 面向调用方的简洁描述
- `details`: 仅在需要时返回，主要用于参数校验错误；未知异常默认不返回内部细节

异常设计：

- 新增统一业务异常基类，例如 `AppException`
- `AppException` 至少包含：`status_code`、`code`、`message`、`details`
- 预留常用子类：
  - `BadRequestException`
  - `NotFoundException`
  - `ConflictException`
  - `UnauthorizedException`
  - `ForbiddenException`
- 注册全局异常处理器：
  - `AppException` -> 使用异常自身的状态码和错误码
  - `RequestValidationError` -> `422` + `VALIDATION_ERROR` + 标准化 `details`
  - `HTTPException` -> 映射成统一错误结构；若未提供业务码，按状态码映射默认语义码
  - `Exception` -> `500` + `INTERNAL_SERVER_ERROR`，不暴露堆栈和内部实现

默认错误码映射建议：

- 400 -> `BAD_REQUEST`
- 401 -> `UNAUTHORIZED`
- 403 -> `FORBIDDEN`
- 404 -> `RESOURCE_NOT_FOUND`
- 409 -> `CONFLICT`
- 422 -> `VALIDATION_ERROR`
- 500 -> `INTERNAL_SERVER_ERROR`

### 3. 日志方案

采用标准库 `logging`，先不引入第三方结构化日志库，保持实现轻量。

日志配置设计：

- 新增日志初始化模块，在应用启动阶段统一配置 root logger
- 日志级别由配置项控制，默认：
  - `local` 环境为 `INFO`
  - 以后可通过环境变量覆盖
- 输出格式为单行文本，推荐字段顺序：
  - 时间
  - 级别
  - logger 名称
  - HTTP 方法
  - 路径
  - 状态码
  - 耗时
  - 核心消息

请求日志行为：

- 新增 HTTP middleware 记录每个请求的访问日志
- 正常请求记录一条完成日志，包含方法、路径、状态码、耗时
- 未捕获异常由异常处理器记录 error 日志
- 4xx 记录为 warning 或 info，5xx 记录为 error
- 健康检查接口可保留日志，但建议降噪处理：
  - 默认记录
  - 实现上预留将 `/health` 排除日志的简单开关

日志边界：

- 不记录敏感信息，如完整请求体、密码、token、cookie
- 当前不引入 `request_id`
- 当前不做 JSON 日志双格式切换，后续如接入容器平台再演进

### 4. 应用集成方式

应用层集成顺序：

- 创建 `FastAPI` 实例
- 初始化日志配置
- 注册异常处理器
- 注册请求日志 middleware
- 挂载路由

建议新增的职责模块：

- `schemas/response.py`: 成功/失败响应模型
- `exceptions.py`: 业务异常定义
- `handlers.py`: 全局异常处理器
- `logging.py` 或 `core/logging.py`: 日志初始化
- `middleware.py`: 请求日志 middleware

尽量保持现有目录风格，不需要一次性引入复杂分层。

## Public Interfaces

新增公共返回接口约定：

- 业务成功响应：`ApiResponse[T]`
- 业务失败响应：`ErrorResponse`
- 业务异常基类：`AppException`

新增可配置项建议：

- `APP_LOG_LEVEL`
- 可选预留 `APP_ACCESS_LOG_ENABLED=true/false`

保持不变：

- 现有 `/api/v1/health` 路径不变
- 现有基础配置加载方式不变
- 现有 `create_app()` 工厂模式不变

## Test Plan

需要补充的测试场景：

- 业务成功响应返回统一壳，包含 `success/code/message/data`
- `/health` 仍返回原生结构，不被统一壳包裹
- 主动抛出 `AppException` 时返回对应状态码和语义错误码
- 请求参数校验失败时返回 `422 + VALIDATION_ERROR + details`
- 未知异常时返回 `500 + INTERNAL_SERVER_ERROR`，且不暴露堆栈
- middleware 能记录请求完成日志
- 5xx 异常路径会产生日志
- 日志中不包含敏感字段

## Assumptions

- 当前项目仍以内部服务/后端接口为主，不需要多语言错误消息
- 当前没有既有前端或外部调用方依赖某种旧响应格式，可以直接建立新规范
- 当前先不引入 `request_id`、Sentry、JSON 结构化日志、统一分页模型
- 统一响应包装优先应用于后续新增业务接口；基础探活接口继续保持轻量即可
